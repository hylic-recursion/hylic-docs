# Information architecture — hylic docs book

*Proposed IA after the 2026-04 crate split + lift/pipeline overhaul.
Guides the rewrite task.*

## Design principles

1. **Reader-first ordering.** Start with *what*, then *how*, then
   *why* (concepts → guides → reference → theory). Deep internals
   (funnel executor) live as their own part, not on the main path.
2. **Code-quoted.** Every non-trivial type or trait definition is
   pulled verbatim from the source via `{{#include}}` — no drift.
3. **Diagrams at boundaries.** Wherever types compose (variance,
   sugar ladders, typestate transitions, lift chains), a DOT
   diagram sits alongside the prose.
4. **Two surfaces acknowledged.** hylic (core — Fold, Graph,
   Executor, Lift atoms) and hylic-pipeline (pipelines + sugars)
   each get dedicated parts. The crate boundary shows in the TOC.
5. **No backwards-compat prose.** If an API changed, the doc
   describes the *current* shape. Git history is where history
   lives.

## Style

- Short sentences where possible. One idea per paragraph.
- Run a code quote early on every non-trivial chapter.
- Diagrams before prose that explains them.
- No marketing language; neutral, load-bearing technical prose.
- Examples always from `docs_examples.rs` (the insta-tested
  reference) or from the test suite — never hand-typed inline.
